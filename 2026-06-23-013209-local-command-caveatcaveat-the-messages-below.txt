  稀疏注意力的"鸡生蛋"难题

  稀疏的目标是把 full attention 的 O(S²) 降下来——只算重要的那些 KV。但马上撞墙:

  ▎ 要判断 query 和某个 KV 重不重要,得先算它俩的注意力分数;要判断所有 KV,就是 
  ▎ full attention,O(S²) 原地复活。

  选择的前提是打分,而打分的代价≈不稀疏。
  这就是所有稀疏注意力绕不开的鸡生蛋问题。NSA 的"块压缩 + top-k 选块"就是为破这
  个局而设计的——它是两级筛:粗筛顺手产出路由,精筛只在重点恢复细节。

  三分支各自在解什么

  ① 块压缩 = 廉价粗筛 + 免费路由器
  - 把 KV 按块(b 个 token)压成 1 个压缩 token,S → S/b 个;query 去 attend 这 S/b
  个,代价 O(S²/b),便宜 b 倍。(nsa.py:57-58 的 mean 压缩;简要版用 mean-pool,真实
  NSA 用小 MLP,但"压缩"这件事一样)
  - 关键副产品:query 对每个压缩块的注意力分数 = 这个块有多重要。这张"块重要性表"
  是算压缩输出时顺手就出来的,不额外花钱。(nsa.py:60 的 sim 一身二职:既喂
  compressed 分支的 softmax,又作为 blk_sim 返回)
  - 但压缩有损:32→1 的平均抹掉了块内细节。所以需要第二级。

  ② top-k 选块 = 精筛恢复细节
  - 拿上一步的块分数,每个 query 选 top-k 个最重要的块(nsa.py:76 的 topk),只对这 
  k 个块的原始(未压缩)token 做精细 attention,代价 O(S·k·b),k 很小(如 4)。
  - 把压缩丢掉的细节,只在重要的少数块补回来。妙处在于:选谁,完全由压缩分支免费提
  供——压缩分支一身二职:全局视野 + 
  选块路由。这就是为什么二者必须成对出现:选择离不开压缩这只"先下的蛋"。

  ③ 滑窗 = 保底局部 + 训练分流
  - 固定 attend 最近 window 个 token(nsa.py:94)。最近的 token
  几乎总相关,不值得"选",直接固定覆盖更省。
  - 更深的原因(论文动机):若不单列局部分支,梯度会诱导压缩/选择去拟合局部模式,污染
  它们本该学的全局/检索能力。三分支分流 → 各司其职。

  query ─┬─→ 压缩: attend S/b 个压缩块 ─→ 粗全局输出 + 块重要性分数 ┐
         │                                          (分数流向选择) ↓
         ├─→ 选择: 用分数选 top-k 块, attend 块内原始 token ─→ 细节输出
         └─→ 滑窗: attend 最近 window 个 token ──────────→ 局部输出
                                            门控加权融合(nsa.py:111) → 输出

  为什么是"块",不是单 token?

  硬件对齐(论文标题里的 Hardware-Aligned)。选"块"→ gather 的是连续内存段 → GPU
  上连续访存,能套 FlashAttention 式的分块在线 softmax。若按单 token 选,gather
  是离散的、访存碎片化,GPU 上反而慢。块粒度就是为了让稀疏在 GPU 
  上真的快——而这恰恰是那 1987 行 triton kernel 在做的事:按选中的块索引去 gather
  KV,再做 flash 式计算。

  ---
  一句话:压缩负责"廉价看全局 + 免费产出选块路由",选择负责"在选中的少数块里恢复细
  节",滑窗兜底局部。压缩是为选择服务的那只先下的蛋。

  这个"为什么"我会写进 08 的 README/notebook。回到刚才的落地选择(薄封装调库 /
